# 統合設計書：介護研修コンプライアンス機能 → あわい屋ZEROS学習PF（主）

> ステータス: 確定スコープ版（2026-06-18）
> 対象: あわい屋ZEROS学習プラットフォーム（Next.js 16 + Supabase）＝**主**
> 統合元: 介護研修管理システム（`care-training-system`）の**ドメイン知識のみ**を移植（コードは原則移植しない）
> 関連: `integration-design-zeros-ai.md`（ZEROS-AI診断エンジン統合）とは別件。同一学習PF上で共存。

---

## §0 確定した大前提（ユーザー確定 2026-06-18）

| # | 前提 | 帰結 |
|---|------|------|
| 0-1 | **学習PFが主**。LMS実行基盤・オーサリング・受講・採点はすべて学習PF | 介護システムのアプリは移植しない |
| 0-2 | **生成AIは不要** | `@anthropic-ai/sdk` 等の導入なし。教材/テスト生成機能は作らない |
| 0-3 | 教材投入（動画・資料PDF・ワークシート・テスト5問）は**学習PFの既存オーサリングで対応済み** | コンテンツ機能の追加は不要 |
| 0-4 | 年間計画の**自動生成は不要**。管理者が「必須研修をいつ実施するか選択」できればよい | AI/ルールエンジンによる計画生成は作らない |
| 0-5 | 残す機能は **A. 必須研修の実施時期選択 ／ ③ 監査対応PDF一括出力 ／ ④ 法改正自動通知** の3点のみ | 本書はこの3機能に限定 |

**結論**: 本統合は **AIもPythonもNeonも持ち込まない、学習PF（Next.js+Supabase）への純粋な追加開発**。介護システムからは「どの研修が法定必須か」「監査に必要な帳票」「法改正の監視先」という**ドメイン知識のみ**を引き継ぐ。

## §1 確定スコープ（3機能）

| 機能 | 内容 | 学習PFの既存資産 | 追加で必要なもの |
|------|------|----------------|----------------|
| **A. 必須研修の実施時期選択** | 管理者が必須研修ごとに「いつ実施するか（実施予定日・締切）」を選択 | `mandatory_courses`（`due_date`付き）＋ `/admin/mandatory` 画面が**既存** | 実施予定日の概念追加（小）＋法定必須研修のチェックリスト提示（任意） |
| **③ 監査対応PDF一括出力** | 必須研修の実施記録・受講状況・テスト結果・（任意で報告書）を監査様式でPDF出力 | `enrollments`/`lesson_progress`（受講・完了） | **`quiz_attempts` 新設（必須）**＋PDF生成基盤 |
| **④ 法改正自動通知** | e-Gov/厚労省から介護関連の改正を検知し管理者へ通知 | `notifications` テーブル＋通知UIが**既存** | `legal_updates` 新設＋定期取得ジョブ |

スコープ外（明示）: 生成AI、AIによる教材/テスト生成、年間計画の自動生成、教材オーサリング機能（すべて不要／既存で充足）。

## §2 機能A：必須研修の実施時期選択

### 2.1 既存資産（実証確認済み）
- テーブル `mandatory_courses`（init.sql:125-132）: `course_id` / `organization_id` / `department_id`(NULL=組織全体) / **`due_date DATE`** / `created_at`。RLSポリシー設定済み（ORG_ADMIN は自組織のみ書込可）。
- 管理画面 `/admin/mandatory`（`MandatoryClientPage.js` / `actions.js`）が既存。

→ **「必須研修を割り当て、締切を設定する」機能は既に動いている**。機能Aは差分のみ。

### 2.2 差分設計（最小）
1. **「実施予定日」の表現**: 介護法定研修は「年度内のいつ実施するか」を管理する。現状 `due_date`（締切）のみなので、必要に応じ `mandatory_courses.scheduled_date DATE`（実施予定日）を追加。
   - 単一日付で足りるなら `due_date` を「実施予定日」として運用し新規列なしも可（§6の決定事項）。
2. **法定必須研修チェックリストの提示（任意）**: 介護システム `trainingMasterData.py` が持つ「事業所種別ごとの法定研修一覧」（訪問介護10／居宅7／共通7）を、`/admin/mandatory` 画面に**参照リスト（チェックリスト）**として表示し、管理者が対応コースを紐付けて必須化しやすくする。
   - 実装はフロント定数 or 参照用 seed テーブル。研修コンテンツ自体は学習PFで作成済みのものを選ぶ。
3. 事業所種別（訪問介護／居宅介護支援）は `organizations` または `departments` の属性として表現し、チェックリストの出し分けに使う（任意）。

### 2.3 成果物
- （必要なら）`scheduled_date` 追加マイグレーション
- `/admin/mandatory` への実施予定日入力・法定研修チェックリスト表示
- 受講者ダッシュボードに「実施予定日／締切」を表示

## §3 機能③：監査対応PDF一括出力

### 3.1 前提となる課題（実証確認済み・要対応）
- 学習PFは**クイズの点数を永続化していない**。`submitQuizAnswers`（受講側 `actions.js`）はその場で採点（80%合格）し、合格時に `lesson_progress` へ完了INSERTするのみ。スコア・受験履歴はDB未保存（UI返却のみ）。`lesson_progress` に score/passed 列なし。
- → 監査PDFに必要な「テスト結果」を出すには、**採点履歴の永続化が必須前提**。

### 3.2 追加テーブル `quiz_attempts`（必須）

| 列 | 型 | 用途 |
|----|----|------|
| id | UUID | PK |
| user_id | UUID FK→users | 受講者 |
| lesson_id | UUID FK→lessons | クイズレッスン |
| score_percent | INT | 得点率 |
| is_passed | BOOLEAN | 合否（80%基準） |
| answers | JSONB | 回答内容（任意・監査詳細用） |
| attempted_at | TIMESTAMPTZ | 受験日時 |

- RLS: 受講者は自分の行のみ、ORG_ADMIN は自組織のみ（既存ポリシー方式に準拠）。
- `submitQuizAnswers` に「採点結果を `quiz_attempts` へINSERT」する処理を追加（合否に関わらず記録）。

### 3.3 PDF生成
- 学習PFはJSのため `@react-pdf/renderer`（日本語フォントは `Font.register()` でNotoSansJP登録）等を採用。
- 集約内容（監査様式・実地指導想定）:
  - 必須研修一覧（実施予定日／締切／対象）
  - 受講状況（`enrollments` / `lesson_progress`：誰がいつ完了したか＝出席記録）
  - テスト結果（`quiz_attempts`：得点・合否）
  - （任意）研修報告書 ※報告書機能を持つ場合のみ。本スコープでは未確定（§6）
- UI: `/admin` に「監査エクスポート」ボタン（年度・組織・期間指定→PDF一括DL）。

## §4 機能④：法改正自動通知

### 4.1 既存資産
- `notifications` テーブル（init.sql:135〜）＋ `/admin/notifications` が既存 → **配信基盤は流用**。

### 4.2 追加要素
- 追加テーブル `legal_updates`: `id` / `source`(e-Gov|mhlw) / `title` / `summary` / `url` / `detected_at` / `status`(new|reviewed)。
- 取得ジョブ: **Vercel Cron**（学習PFは既にVercel/Next.js）から定期実行 → e-Gov法令API v2・厚労省RSSを取得 → 介護関連をフィルタ → `legal_updates` へINSERT → 該当を `notifications` へ連携してORG_ADMINに通知。
- UI: `/admin` ダッシュボードに法改正セクション（一覧・既読化）。

## §5 実装Phase計画

| Phase | 目的 | 主な作業 | 検証 |
|-------|------|---------|------|
| **0** | 設計確定 | 本書レビュー・§6の決定 | ユーザー承認 |
| **1** | 機能A（実施時期選択） | （必要なら）`scheduled_date`追加・`/admin/mandatory`拡張・法定研修チェックリスト | 必須研修を割当→実施予定日設定→受講者に表示 のE2E |
| **2** | 機能③（監査PDF） | `quiz_attempts`新設・`submitQuizAnswers`に保存追加・PDF生成・監査エクスポートUI | 受講→テスト→完了→PDFに実施/出席/点数が出る のE2E |
| **3** | 機能④（法改正通知） | `legal_updates`新設・Vercel Cron取得ジョブ・`notifications`連携・UI | 取得→検知→通知→既読 のE2E |

各Phaseは独立。優先度の高いものから着手可能。**機能③のみクイズ採点保存（Phase2前段）が前提**となる点に注意。

## §6 ユーザーの決定待ち事項

1. **機能Aの実施日**: 「実施予定日」を新規列（`scheduled_date`）で持つか、既存 `due_date` を実施日として運用するか。
2. **法定研修チェックリスト**: `/admin/mandatory` に法定必須研修の参照リスト（訪問介護/居宅/共通）を表示する機能を入れるか（任意）。
3. **監査PDFに研修報告書を含めるか**: 含める場合、報告書の作成元（手入力 or 学習PFの別機能）を決める必要。本スコープでは「実施記録・受講状況・テスト結果」までを既定とする。
4. **着手Phase順**: A → ③ → ④ の順か、③（監査）を最優先にするか。
5. **介護システム（現リポジトリ）の扱い**: ドメイン知識移植後に廃止するか保管するか。

## §7 リスク・留意点

| 項目 | 影響 | 対応 |
|------|------|------|
| クイズ採点が未永続（実証済み） | 監査PDFのテスト結果源が現状無い | `quiz_attempts` 新設＋保存処理（Phase2の必須前段） |
| 監査様式の法令準拠 | 実地指導で不備リスク | 厚労省ガイドライン・自治体様式に合わせて項目を確定（Phase2設計時） |
| 法改正フィルタ精度 | 介護無関係の通知混入 | キーワード/法令番号でフィルタ。`status`で人手レビュー前提 |
| 多テナントRLS | データ越境 | 新規テーブルは既存RLS方式（org_id＋role）を踏襲 |
| Vercel Cron頻度 | 取得漏れ/過剰 | 日次程度で十分（法改正は低頻度） |
